# Project Auto Bot 스팩

프로젝트의 운영 정보 및 작업 정보를 획득하고 실시간으로 프로젝트를 수정하는 봇을 만듭니다.

해당 봇의 이름을 **`orangios`** 으로 설정합니다.

---

## 스팩 1: 지라 티켓으로부터 작업 획득

### 시나리오

1. 지라 티켓을 생성합니다.
2. 티켓 내 진행할 작업에 대한 명세를 자세히 입력합니다.
3. 티켓의 작업을 진행하는 대상이 `orangios`임을 명시합니다. (진행자 혹은 태그)
4. JIRA MCP를 사용하여 작업 내역을 획득하고 실제로 작업을 수행합니다.

### 권한 모델

- `orangios`는 사실상 JIRA 앱처럼 동작합니다. 즉, JIRA 티켓을 생성/관리/할당할 수 있는 권한을 가진 사용자라면 누구나 `orangios`에게 작업을 지시할 수 있습니다.
- 따라서 "누가 봇을 부릴 수 있는가"의 보안 경계는 고객사의 JIRA 권한 설정에 위임되며, 이 영역의 통제는 사용자(고객)의 책임입니다.

---

## 스팩 2: 크래쉬 애널리틱스로부터 작업 획득

### 시나리오

1. 크래쉬 애널리틱스의 훅(webhook) 이벤트를 통해 급증하는 크래쉬 이벤트를 획득합니다. (필요 시 폴링 방식도 보조적으로 제공할 수 있습니다.)
2. 해당 이벤트를 판단하고 자동으로 JIRA 티켓화 합니다.
3. 사용자는 해당 이벤트를 JIRA를 통해 습득하고 직접 해결하던지 아니면 AI에게 맡깁니다.

### 비고

- "급증"의 정의(증가율, 절대 건수, 영향 사용자 수 등)는 MVP 단계에서는 사용자가 커스텀하게 정의하도록 하며, 정교한 자동 판단 로직은 추후 결정합니다.

---

## 세부 스팩 1: 깃허브 PR 관련

- 생성한 풀리퀘스트의 리뷰어는 기본적으로 `CODEOWNERS` 규칙을 따라 지정합니다.
- 옵션으로, 수정한 코드의 이전 blame을 읽어 해당 작성자에게 리뷰를 요청할 수 있습니다. 이는 리뷰 피로를 줄이는 것이 주 목적이며, 켜고 끌 수 있는 설정으로 제공합니다.
- `@orangios`로 태그 및 코멘트를 남기는 경우 해당 리뷰를 반영하여 새롭게 커밋을 남기거나 사용자에게 의견을 알립니다.
- 병합은 GitHub의 브랜치 보호 규칙에 위임합니다. 즉 `orangios`가 임의로 머지하지 않고, 저장소에 설정된 규칙(필수 CI 통과, 필수 승인 수 등)이 충족되면 병합을 진행합니다.
- 서비스는 사용자에게 GitHub 브랜치 보호 규칙 설정(최소 1개 이상의 Approve 시 병합 가능 등)을 권장하는 안내를 제공합니다.

## 아키텍처 스팩 1: 시스템 아키텍처

- AWS에서 클로드 코드를 돌리기 위한 서버를 운영합니다. spot 대신 장기적으로 유지되는 프로비저닝을 사용하여 비용을 최소화하는 방향으로 합니다. (구체적 인스턴스 타입은 추후 결정)
- 지라, 크래쉬 애널리틱스의 훅 이벤트가 발생하면 해당 이벤트를 서버로 전송합니다. 이벤트 수신부와 처리부는 큐를 통해 분리하여 유실·중복을 방지합니다.
- 서버에서 이벤트를 처리합니다. (깃허브 작업, PR 생성 및 대응) 작업의 복잡도 및 효율도에 따라 멀티 서버를 가질지는 추후에 결정합니다.

### 작업 실행 모델

- 큐(SQS 등)를 작업 전달 계층으로 둡니다. 큐는 작업 실행이 아니라 다음을 담당합니다.
  1. 이벤트 버퍼링(폭주 흡수)
  2. 재시도
  3. 멱등성(중복 이벤트 차단)
  4. 백프레셔
- 장기 유지되는 워커 풀이 큐에서 작업을 하나씩 꺼내 Claude Code 세션을 끝까지 수행합니다. (매 작업마다 일회성 인스턴스를 띄우지 않습니다.)
- 워커는 레포별 워크스페이스를 디스크에 캐시하여, 다음 작업 시 클론·의존성 재설치 없이 `git fetch`/`reset`만으로 재사용합니다. → 무거운/장시간 작업의 콜드 스타트 비용을 회피.
- 작업 실행 부분은 작업당 컨테이너로 격리하여, 외부 입력(JIRA 티켓 본문·크래시 로그)을 통한 프롬프트 인젝션의 영향 범위를 제한합니다.
- 동시 처리량은 워커 풀 크기 × 작업당 리소스로 조절하며, 작업 자체를 잘게 분할할 필요는 없습니다.
